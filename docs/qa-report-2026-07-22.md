## High-impact local changes verified

### Connector and Zoom truth

- The backend and frontend catalog now expose only the supported Gmail, Drive, Notion, and Slack ingestion sources.
- Unsupported OAuth is rejected. Existing unsupported ingestion attempts persist a failed Sync Run/account error instead of pretending to work.
- Connector foreign keys now use a shared, concurrency-safe connector creation path on fresh databases.
- Remote connector logos were removed from the critical UI path and unsupported reconnect/sync controls are hidden.
- Zoom is hard-disabled: the route is hidden/404, capability is false, mock/download/transcription worker fallbacks were removed, and legacy Zoom environment variables cannot enable it or block startup.

### Composio, cloud provenance, and sync truth

- The merged direct Composio tool-calling path is production-default-off. When explicitly enabled, it is admin-only and treats unclassified live provider data as red: cloud use requires the workspace's red-tier policy to allow it.
- Direct-tool and live-context answers do not inherit unrelated RAG citations. Action/delivery policy receives red provenance, so unclassified provider content cannot be passed off as a normal-tier cited answer.
- Tool results are labelled untrusted, question/history/tool-result sizes are bounded, connected-account email identities are omitted from model context, and only org-scoped active/configured connectors are described as connected.
- Provider-returned download URLs now fail closed unless they are HTTPS, match an operator allowlist, and resolve only to global addresses. Redirects are not followed. The allowlist is intentionally blank by default, so remote export/media downloads require explicit staging validation and configuration.
- Embedding requests are capped at 96 chunks, vector failures produce a visible `partial` Sync Run instead of false success, and unsupported legacy ingestion attempts create a durable failed Sync Run.
- Remaining data-model risks are explicit: source-document IDs are still provider-global rather than organization-namespaced, and `(org_id, connector_key)` uniqueness is not enforced for connector accounts. Both require migrations and PostgreSQL concurrency rehearsal.

### Scheduler and deployment truth

- Render defines a paid execution worker and Celery beat for recurring automations. The worker consumes the `execute` queue.
- Beat emits `scheduler_heartbeat` every 60 seconds through that queue; the worker writes a 180-second Redis heartbeat.
- `/capabilities`, backend create/update validation, and the UI fail closed unless the complete beat-to-queue-to-worker path is fresh.
- External cron and Celery beat are now mutually exclusive configuration modes. Render and Compose select one worker/beat scheduler and hide the external-cron route; startup rejects a configuration that enables both.
- Due selection still has no database-backed per-automation lease. Concurrent manual/token/scheduled execution can therefore overlap before `last_run_at` is committed; an atomic PostgreSQL claim remains required.
- This does not move Composio/provider ingestion off API `BackgroundTask`; docs and Compose now say so explicitly.

### Dashboard, Sync Runs, and notifications

- Dashboard pending actions/decisions, recent decisions, and connector records now use persisted data and expose an `as_of` timestamp that is visible in the UI.
- Sync Runs supports cursor pagination, server totals/aggregates, and Load More.
- Notifications now expose stable tied-timestamp paging, total/unread counts, explicit fail-closed individual and Mark all read mutation, a sidebar unread badge, polling, type-aware thread deep links, neutral unknown-type rendering, and exclusive loading/error/empty states. Browser coverage includes a failed-then-retried Mark all read flow.
- Private-thread mentions no longer notify people who cannot open the thread. Document-share notices are removed when access is revoked or reset deletes the underlying documents.

### Saved artifacts and timestamp semantics

- Saved artifacts no longer collapse sensitive Ask/SQL output to organization-wide member access. Members can list/delete only their own rows, current admins can manage all rows, and legacy ownerless rows are admin-only.
- Delete is atomic, concealed across authorization boundaries, and retry-safe. Stable `(created_at,id)` paging reaches histories beyond the old 200-row cap.
- New writes use the existing OpenUI artifact schema. Malformed legacy JSON renders an unsupported-artifact warning while Export/Delete remain reachable instead of crashing the page.
- Database timestamps now serialize through one UTC helper with an explicit `Z`; naive database values are treated as UTC and aware values are normalized to the same instant.

### Team lifecycle and authorization

- Admins can revoke invites, rename/delete departments, unassign members, and inspect removal impacts.
- Department foreign keys use deliberate `RESTRICT`/`SET NULL` behavior through migration `0026` with cleanup.
- Migration `0031` adds composite `(org_id, department_id)` foreign keys for users, invites, and source documents, cleans invalid legacy references, and makes the database enforce same-workspace ownership. SQLite migration/downgrade/round-trip coverage passes; live PostgreSQL lock duration and DDL remain a staging gate.
- Invite acceptance/revocation and last-admin mutation paths use compare-and-set/organization locks to close races.
- Member removal supports an atomic same-organization successor transfer for automations, threads, and workflow runs; transferred automations are paused and tokens cleared.
- Saved-artifact attribution preserves the creator name while detaching the removed creator ID.
- Removal fails closed for owned uploads, document grants, SQL OrgMemory, AskExchange, pending connector actions, and invalid cross-org successors.
- The UI previews impacts/blockers, requires explicit successor selection, and explains sole-admin account deletion remediation.
- External-only Supermemory user containers still cannot be completely enumerated and purged.

### Invitation/OAuth binding and shared-demo safety

- Invitation links now carry only an opaque token in the login URL fragment. The login page immediately scrubs that fragment and retains the token only in component state, keeping it and the invited email out of normal URL/query logs.
- Invitation-backed Google login uses a top-level form POST with a strict single-field, 512-byte form parser. A signed HttpOnly OAuth-state cookie binds the exact token; callback acceptance atomically consumes that exact fresh invite for the canonical verified email. Wrong, revoked, expired, replayed, mismatched, and competing cross-organization tokens fail closed without silently provisioning an ordinary organization.
- Ordinary Google sign-in remains a query-free GET. The Content Security Policy permits only the application/API form targets and the exact Google Accounts origin needed for the browser redirect.
- Non-local `POST /orgs` now returns 403, and production onboarding redirects to `/login` instead of exposing public email/workspace creation. Organization provisioning stays on the verified Google identity path; deployed environment configuration and provisioning audit telemetry remain unproven.
- The shared demo is now read-only at the API boundary for both anonymous demo headers and a valid demo-admin JWT. ### Ask durability

- Migration `0027` adds durable Ask-exchange reservations before model execution.
- Concurrent requests with the same request ID wait/replay; a different payload with that ID conflicts.
- The answer is durably marked answered before the user message, assistant message, mentions, thread state, and completed status are committed atomically.
- The API distinguishes inaccessible thread (409), still processing/persistence failure (503 with `Retry-After`), and saved results.
- The client retries once with the same request ID and restores citations, actions, context status, model/latency metadata, artifacts, and feedback during hydration.

### Persistence, action dismissal, and session truth

- Dismissal is now a server-owned `proposed -> dismissed` transition, authorized to the proposer or a current workspace admin, idempotent after reload, and atomically mutually exclusive with confirmation. A store outage before the claim remains visibly retryable; unknown transport/provider outcomes remain terminal to avoid duplicate side effects.
- Per-action busy state prevents one pending action card from re-enabling another, and status/error changes are announced to assistive technology. Public action failures use stable generic messages while detailed provider exceptions remain server logs.
- Workflow list/detail storage failures return generic 503 responses, and a workflow run is not reported successful when its post-model persistence fails. Workflow transcripts and automation names/prompts/delivery targets now have explicit type and length bounds.
- Document upload persistence failure rolls back and returns 503 before Qdrant is called, preventing orphan vectors. Post-persistence vector/access lag returns stable public codes without raw exception strings.
- Integration and Decision list storage outages return 503 instead of masquerading as empty workspaces. Graph access-map storage outages likewise render an explicit retry state rather than an empty graph.
- Successful sign-out-all now expires the current HttpOnly cookie as well as revoking token generations. If global revocation fails but local cookie clearing succeeds, Login warns that other sessions may remain; if both calls fail, Settings retains the session and offers a retry.

### Slack replay, external-execution safety, and canonical identity

- Migration `0028` adds a durable signed-Slack-request receipt keyed by a database primary key. Mapping is resolved before the atomic claim, so only one normal worker can claim a request; focused SQLite/unit evidence passes, but no real PostgreSQL race was exercised.
- Slack response work still crosses into a FastAPI/Starlette `BackgroundTask`. That handoff is not durable, so a process crash after acknowledgement can lose the answer even though replay spend is guarded.
- Proposed actions require an atomic transition from `proposed` plus a fail-closed 24-hour cutoff. The branch now persists a connector-action execution record before the provider call, derives a stable key, sends Slack's native `client_msg_id`, retries only definite preflight failures, and quarantines provider/ack uncertainty as terminal `outcome_unknown` instead of blindly replaying a possible duplicate.

Payloads are bounded to 64 KiB; records use seven-day retention with bounded opportunistic cleanup.

### Cost controls, proxy identity, and Data Routing

- Twenty expensive endpoints are assigned to eight cost classes covering Ask/action confirmation and dismissal, workflows, automation execution/triggers, document uploads, evals, Composio/integration operations, search, SQL, and verified Slack tenants.
- Current per-tenant/client/route budgets are AI 20/minute, provider action 10/minute, workflow 10/hour, eval 3/hour, ingest 10/hour, OAuth 10/hour, SQL plan/schema 20/hour, and SQL execute 30/minute. These are implementation defaults, not an approved aggregate-tenant spend policy.

Render uses the documented first-forwarded-address mode; malformed leading entries are skipped rather than making that mode fail closed.
- Safe reads remain unthrottled. The budgets are static code constants and composite tenant/client/route limits, not an aggregate tenant quota; distributed clients can multiply the tenant's total allowance. Aggregate caps are intentionally not guessed: product/operations must choose tenant caps/windows and request-count versus weighted-spend semantics. PATCH carries the last confirmed `expected_routing`; the backend locks and compares the stored row and returns 409 for a stale or omitted expectation when the stored policy is valid. A missing/null expectation is accepted only to recover malformed stored policy to deny-all. Real PostgreSQL two-session lock behavior is still a staging check.

## Residual security and privacy risks

- Signed Slack requests now use durable primary-key receipts, but the cross-worker race was not exercised against real PostgreSQL and the response `BackgroundTask` handoff is not durable; a crash after acknowledgement can still drop the answer.
- Composio capability/read classification still relies partly on toolkit/slug heuristics instead of authoritative provider metadata.
- Direct Composio tool calling remains production-default-off. Live/provider results have only worst-case red provenance, not document-level citations/ACLs; the exact remote-download host allowlist is blank until staging establishes the provider's trusted temporary-download domains.
- Provider-global source-document IDs cause a legitimate second organization ingesting the same external object to be skipped, while connector accounts lack an enforced `(org_id, connector_key)` unique key. Both need data migration and real PostgreSQL concurrency tests.
- Scheduler topology now prevents cron and beat from being enabled together, but manual/token/scheduled runs still lack one atomic per-automation database claim and can overlap before `last_run_at` is recorded.
- Workflow actions conservatively quarantine ambiguous results, but only Slack has a native stable provider key. Freshdesk and other providers need lookup/reconciliation support, and there is no reconciliation worker or operator UI.
- The connector-action record is durable pre-call evidence, not a transactional delivery worker/outbox; real PostgreSQL contention and live provider commit/timeout behavior were not exercised.
- External-trigger idempotency records expire after seven days, after which reuse of the same key can start new work. That contract needs explicit product/operational policy.
