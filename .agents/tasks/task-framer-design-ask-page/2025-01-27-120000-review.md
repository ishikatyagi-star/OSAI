# Framer design system applied to Ask OSAI page

The Ask page and its sub-components (message bubbles, citation chips, action cards, workflow cards) have been restyled to match the Framer DESIGN.md spec. The approach is CSS-first: globals.css classes get updated token values (font sizes, tracking, radii, shadows), while the React components swap Tailwind presets for explicit `var(--*)` references. Gradient spotlight cards are the marquee change — three radial-gradient variants (violet, magenta, orange) replace flat charcoal cards. A new `framer/DESIGN.md` reference file was generated by the `getdesign` CLI.

**Watch for:** gradient variables reference existing definitions but the fallback story is absent — if `--grad-violet` is ever undefined the card degrades to transparent (confirmed). The `.ask-title` uses `font-weight: 600` where the DESIGN.md spec calls for 500 (confirmed). Message bubble radius uses Tailwind's `rounded-xl` (12px) rather than the DESIGN.md `rounded.xl` (20px) — a quiet spec divergence (confirmed).

## High-level view

Typography was bumped across all `.ask-*` classes to match DESIGN.md tokens — 62px display-lg for the title with -3.1px tracking, 18px body-lg subtext, 13px caption for section labels, 12px micro for disclaimers. One exception: the title uses weight 600 where the Framer spec dictates 500 for display-lg.

The workflow cards shed their border-based charcoal treatment for borderless gradient spotlight cards. Each card receives a named gradient class (`--violet`, `--magenta`, `--orange`) applied via a new `gradient` field on the `ASK_MODES` data array. The gradients reference `--grad-*` variables defined at `:root` in globals.css, but without CSS fallback values in the `radial-gradient()` expressions.

Sub-components (message-bubble, citation-chip, action-card) move from Tailwind's `bg-card` / `border-border` pattern to explicit `var(--bg-surface)` / `var(--border)` references. These use Tailwind arbitrary-value syntax rather than scoped CSS classes, creating a second styling convention within the same feature. Radii shift from `rounded-2xl` (16px) to `rounded-xl` (12px) for bubbles — but Tailwind's `rounded-xl` maps to 12px while DESIGN.md's `rounded.xl` token is 20px, so the bubbles are tighter than the spec intends.

## Title typography: weight divergence from spec

The `.ask-title` class sets `font-weight: 600` while DESIGN.md's `display-lg` token specifies weight 500. The project uses Inter Variable (not GT Walsheim), so bumping weight to compensate for font personality differences is plausible — but it should be documented or the spec updated, because a future developer pulling values from DESIGN.md will get a different result.

```css
/* globals.css — current */
.ask-title {
  font-size: clamp(38px, 5vw, 62px);
  font-weight: 600;         /* ← spec says 500 */
  letter-spacing: -3.1px;
  line-height: 1.0;
}
```

## Gradient fallback gap

The spotlight gradient declarations use the pattern:

```css
.ask-mode-card--violet {
  background: radial-gradient(120% 120% at 15% 0%, #8b6bff 0%, var(--grad-violet) 45%, #1a0c40 100%);
}
```

`--grad-violet` is defined at `:root` scope in globals.css (line 52), so this works today. But unlike the adjacent color references that use raw hex, these depend on a variable without a CSS fallback. If the variable definition is ever moved or split out, the 45% stop becomes `transparent`, producing a broken gradient with no visual signal. Adding `var(--grad-violet, #6a4cf5)` makes this self-healing.

## Bubble radius: Tailwind token vs. DESIGN.md token

Message bubbles use `rounded-xl` which Tailwind v4 resolves to `border-radius: 0.75rem` (12px). DESIGN.md's `rounded.xl` token is 20px. The mismatch creates a subtly tighter bubble than the spec intends. The fix is either `rounded-[20px]` (exact spec) or `rounded-2xl` (16px, Tailwind's next step up). The same issue doesn't affect the workflow cards because those use the CSS class `.ask-mode-card` with an explicit `border-radius: 20px`.

## Mixed styling idiom: CSS classes vs. Tailwind arbitrary values

The Ask page uses scoped CSS classes (`.ask-composer`, `.ask-mode-card`) in globals.css. The sub-components use Tailwind arbitrary-value syntax:

```tsx
// citation-chip.tsx
"rounded-full border border-[var(--border)] bg-[var(--bg-surface)] px-2.5 py-1 text-[13px]..."

// action-card.tsx
"rounded-[15px] border bg-[var(--bg-surface)] p-4"
```

This means the Ask feature has two styling strategies: globals.css classes for the page, inline Tailwind for the components. A developer looking for "where is the citation chip styled?" has to check both places. As the Framer treatment rolls out to the rest of the app, converging on one approach will reduce this friction.
