# UI/UX audit polish: 9 visual fixes across sidebar, search, badges, and layout

Nine CSS-level fixes addressing a senior engineer's UI/UX audit. The changes correct a CSS variable shadowing bug on active sidebar icons, improve contrast on badges and action links, bump spacing/widths on table columns and tabs, increase search bar prominence, add bottom padding to gradient cards, and add a missing legend entry to the org graph. All changes are visual-only (CSS and minimal TSX), and the build/typecheck pass cleanly.

**Watch for:** The search bar's hover rule is now a no-op after the default styles were elevated to match (confirmed). A `focus-within` rule described in the implementation plan was never added, leaving keyboard-focus undiscoverable (confirmed). The dismiss action's hover opacity conflicts with the new color upgrade (likely).

## High-level view

The active sidebar icon fix is the most structurally interesting change: `var(--accent)` in `globals.css` resolves to `#0099ff` in isolation, but `theme.css` redefines `--accent` to `#1c1c1c` for shadcn/ui components, making the icon invisible on the same-colored elevated background. Switching to `var(--text-primary)` bypasses the conflict entirely.

The search bar got two of its three planned improvements — elevated background and stronger border — but the `focus-within` rule that would add a blue ring on keyboard focus was omitted. The hover rule was not updated to account for the new defaults and is now dead code.

The badge, dismiss-link, and spacing fixes are straightforward value tweaks. The org graph legend addition is minimal inline JSX that matches the existing pattern. The dismiss hover has a minor opacity interaction worth noting.

## Active sidebar icon: CSS variable shadowing

The original rule `.sidebar-nav-item.active .nav-icon { color: var(--accent); }` relied on `--accent` resolving to the global `#0099ff`. Because `theme.css` redefines `--accent: #1c1c1c` on `:root` for shadcn/ui's "accent" design token, cascading order gave the icon the same color as the `--bg-elevated` background — invisible.

The fix uses `var(--text-primary)` (white), which sidesteps the variable conflict and matches the parent rule on `.sidebar-nav-item.active` which already sets `color: var(--text-primary)`. If the intent was a blue highlight rather than white, the proper approach would be a dedicated variable like `--accent-brand` that theme.css doesn't shadow.

## Search bar: dead hover and missing focus state

The diff promotes the search bar from `--bg-surface` / `--border` to `--bg-elevated` / `--border-hover`. The `.topbar-search:hover` rule was left unchanged:

```css
.topbar-search:hover { border-color: var(--border-hover); background: var(--bg-elevated); }
```

Both properties now match the element's default state, so hovering produces no visual change. The hover should either be removed or stepped up further (e.g., `background: var(--bg-hover)` or `border-color: var(--accent-ring)`).

More significantly, the implementation plan described a `focus-within` rule:

```css
.topbar-search:focus-within { border-color: var(--accent); box-shadow: 0 0 0 2px var(--accent-dim); }
```

This rule does not exist in the committed CSS. Without it, keyboard navigation into the search bar has no visible ring, failing accessibility expectations for focus indicators.

## Dismiss link: opacity conflict on hover

The dismiss action moves from `--text-muted` to `--text-secondary`, and the new `:hover` rule sets `color: var(--text-primary)`. The parent hover rule `.attention-card-action:hover { opacity: 0.7; }` still applies, which will dim the element to 70% opacity on hover even as the color brightens. The visual result is a muted white instead of full white. If full white on hover is the intent, `opacity: 1` should be added to `.attention-card-action.dismiss:hover`.

## Layout spacing: tabs padding omission

The `tabs-underline` margin-bottom moves from 22px to 24px. The plan also called for `padding-bottom: 2px` which was not added; without it, the active tab's 2px border-bottom still sits tight against the content below since the `margin-bottom: -1px` on the trigger overlaps the container's border.
